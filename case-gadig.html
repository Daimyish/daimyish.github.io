Deadshot & King Fisher

GADIG Team Projects 2024–2025 QA & Programming

The Problem

GADIG (Game Analysis and Design Interest Group) at George Mason produces collaborative student games each semester. These are large-team projects where reliable QA and debugging become critical — a single uncaught crash can derail a demo build for the entire team.

My Role

QA and Programming Support across two projects: Deadshot (a top-down shooter) and King Fisher (a multiplayer fishing-themed party game). I focused on stability testing, edge-case identification, and bug documentation as well as level creation and layout testing.

Process

  • Performed crash testing by systematically pushing game systems to their limits — rapid input sequences, boundary collisions, pause/unpause during transitions, and resource exhaustion scenarios.
  • Identified and documented edge-case bugs with detailed reproduction steps: exact inputs, system state, expected behavior, actual behavior, and severity rating. This let developers reproduce issues on first attempt.
  • Submitted structured bug reports through the team's tracking system, categorizing by system (physics, UI, audio, gameplay logic) and priority (blocker, major, minor, cosmetic).
  • Contributed programming fixes for minor issues when bandwidth allowed, working within the existing codebase conventions and submitting through code review.

Outcome

Both projects shipped their semester builds on schedule. The structured QA approach I brought helped the teams catch several blocker-level bugs before demo day that would have caused visible crashes in front of faculty and industry guests.

Reflection

Working on someone else's code taught me more about readable architecture than writing my own ever did. When you're debugging a system you didn't build, you viscerally understand why naming conventions, comment headers, and consistent patterns matter. I carry that into every project now.
